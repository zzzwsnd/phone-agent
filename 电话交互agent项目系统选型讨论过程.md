# 电话交互 Agent 项目系统选型讨论过程

## 讨论日期：2026-05-02

---

## 一、需求背景

项目目标：为工业园区设计语音 AI 访客登记系统，替代保安人工问询流程。

核心场景：访客车辆到达 → 拨打号码 → AI 接听 → 自然对话采集信息（车牌、来访单位、手机号、事由）→ 推送至保安微信 → 保安确认放行。

关键约束：
- 25 秒内完成（接通到微信消息发出）
- 3 轮对话约 15 秒
- Human Friendly，像真人门卫一样自然简洁

加分项：回访识别、多路并发、门卫查询 Agent

---

## 二、架构讨论


### 2.1 LangChain：不需要

不管选传统管线还是端到端，LangChain 在此场景下只是多余抽象层：
- 传统管线：LiveKit AgentSession 已管理 STT→LLM→TTS 循环
- 端到端：语音模型自管理对话循环

**结论**：不使用 LangChain。

### 2.2 回访识别：对话前注入，非对话中处理

**核心疑问**：回访需要长记忆，长记忆拖慢 LLM 推理，是否要为实时性牺牲回访能力？

**结论**：这是个伪命题。回访识别的正确做法是对话前注入：
1. 电话呼入时拿到主叫号码
2. 查数据库命中历史记录 → 生成简短摘要
3. 摘要注入 system prompt
4. LLM 只需读几十个字的上下文，不是处理长记忆

LLM 的工作是"读摘要、生成自然语言"，不是"从历史记录中推理出这是回访者"。

**待确认**：回访匹配键的设计——主叫号码？换号怎么办？车牌号是先采后查，有鸡生蛋问题。

### 2.3 呼叫方向：外呼 → 呼入

现有代码是外呼（Agent 主动拨出），需求是呼入（访客打进来）。SIP 配置逻辑、房间创建时机、流程起点完全不同。需确认 LiveKit SIP inbound 接入方式。

## 三、核心选型讨论：传统管线 vs 端到端语音

### 方案 A：STT → LLM → TTS（传统管线）

```
用户语音 → STT(流式) → 文本LLM(流式) → TTS(流式) → 用户听到
```

**优势**：
- 延迟可控：流式管线首字响应 1-2 秒，25 秒做 3 轮够用
- 成本低：专用小模型各做一件事，STT/TTS 单价低，文本 LLM 处理几百 token 极便宜
- 可控性强：system prompt 精确约束输出格式和行为，3 轮内完成采集容易保证
- 模型选择灵活：DeepSeek/Qwen/OpenAI 随意切换

**劣势**：
- 丢失语音信息：语气、犹豫、情绪被文字压平
- 多组件集成调试成本
- 对话自然度不如端到端

### 方案 B：Speech-to-Speech（端到端语音模型）

```
用户语音 → 语音模型 → 用户听到
```

**优势**：
- 对话自然度高：能处理打断、语气词、情绪感知
- 架构简单：少三层集成
- 信息保真：保留语音中的非语言信息

**劣势**：
- 模型选择极少：成熟方案只有 GPT-4o Realtime API，国内生态弱
- 成本高：音频 token 比文本 token 贵几十倍，三个专用小模型比一个通用大模型便宜
- Tool Calling 调试体验差
- 可控性弱：prompt 约束力不如文本 LLM，容易出现话多、跑题
- 绑定 OpenAI：网络延迟、合规风险

### 成本对比核心逻辑

直觉"三个模型 > 一个模型"是错的。原因：
- STT/TTS 是专用模型，非通用 LLM，单价极低
- 文本 LLM 处理 token 量极少（3 轮约 200-300 token），DeepSeek 上几厘钱
- 音频 token 信息密度远低于文本，同样一句话音频 token 数是文本几十倍
- **三个专用小模型各做一件事，比一个通用大模型做所有事，便宜**

### 当前倾向

用户表示对话自然度"不需要太花哨"，传统管线够用。方案 A 更稳妥。国内端到端语音模型待调研。

### 6.1 回访匹配键设计：混合方案

**结论**：主叫号码为主键，其他匹配方式走检索。第一版只做号码匹配，检索方案保留设计但暂不实现。

#### 主路径：号码预注入（零延迟）

```
呼入 → 拿主叫号码 → 查 visitor 表(phone) → 命中 → 生成摘要注入 prompt
```

- 号码匹配覆盖绝大多数回访场景（访客换号概率低）
- 换号场景当新访客处理，体验无硬伤
- LLM 零负担，只读上下文

#### 辅助路径：对话中检索（tool call，V2）

访客在对话中提到车牌号或人名，LLM 通过 tool call 查数据库动态获取历史。

- 匹配维度灵活（车牌、来访单位、人名等）
- 增加约 1-2 秒 tool call 延迟，仅在号码未命中时触发
- 检索结果注入上下文，后续对话自然衔接

#### 分层策略

| 场景 | 路径 | 延迟 | 版本 |
|------|------|------|------|
| 号码命中 | 预注入摘要 | 零 | V1 |
| 号码未命中，访客提到车牌/人名 | tool call 检索 | +1-2s | V2 |
| 换号且未提及其他关联信息 | 当新访客处理 | 无 | V1 |

### 6.2 SIP 呼入方案：LiveKit SIP + Self-hosted Agent

**结论**：SIP 使用 LiveKit 框架提供的电话 Room 服务，Agent 采用 Self-hosted 部署。外呼代码先保留，支持双模式共存。

#### 为什么选 LiveKit SIP

- **架构一致性**：项目已在 LiveKit Agents 上，无需引入第三方 SIP 网关的适配层
- **避免第三方鉴权复杂度**：其他 SIP 服务商需要单点登录校验流程，LiveKit SIP trunk 配一次长期有效
- **Room 模型天然匹配**：一通电话一个 Room，状态隔离天然成立

第三方 SIP 方案对比：

```
第三方SIP网关 → 适配/转接 → LiveKit Room → Agent   （多一层故障点）
LiveKit SIP Trunk → 自动创建Room → Agent dispatch   （直通）
```

#### 为什么选 Self-hosted 而非 Cloud Agents

| | Cloud Agents | Self-hosted |
|---|---|---|
| 部署 | 推代码即可 | 自己维护服务器 |
| 延迟 | Agent 逻辑在海外执行 | Agent 逻辑在国内执行，更利于 25 秒时限 |
| 灵活性 | 受 Cloud Agents 运行时限制 | 完全自由，快速迭代 |
| 成本 | 按分钟计费 | 服务器固定成本，量级不大时更便宜 |

选择 Self-hosted 的核心理由：
1. **延迟**：Agent 处理环节在国内，对 25 秒时限场景每 100ms 都要省
2. **灵活性**：项目在快速迭代期，改了就跑不受 Cloud 运行时限制
3. **成本**：量级不大，一台国内云服务器（2C4G）比按分钟计费便宜

#### 部署架构

```
访客拨打号码
    ↓
LiveKit SIP Trunk（LiveKit Cloud 侧）
    ↓ 自动创建 Room
Agent Worker（国内服务器，Self-hosted）
    ↓ dispatch 加入 Room
AgentSession 启动对话
    ↓ STT → LLM → TTS
访客听到语音
```

与 LiveKit Cloud 之间只有信令和媒体转发，SIP 接入和 Room 管理由 LiveKit 侧负责。

#### 注意事项

- **中国大陆号码 SIP Trunk 可用性**：需确认 LiveKit SIP trunk 是否支持中国号码 inbound，建议先打一通测试
- **网络稳定性**：Self-hosted Agent Worker 需保持到 LiveKit Cloud 的长连接，建议选有优质国际带宽的云厂商（阿里云/AWS 中国出海线路）
- **呼入改造工作量**：主要是删除 `dial_sip` 逻辑、入口参数来源从 metadata 改为 SIP participant attributes、SIP trunk 配置从 outbound 改 inbound，架构不需重构

### 6.3 STT/TTS 选型：中文语音模型替换

**结论**：当前代码硬编码的 Deepgram（STT）和 Cartesia（TTS）是原牙科外呼项目遗留，不适合中文场景，需替换为国内方案。

#### 问题分析

当前代码中的配置：
```python
stt=deepgram.STT(),      # 英文优先，中文识别差
tts=cartesia.TTS(),      # 英文语音，中文合成差
vad=silero.VAD.load(),   # VAD 语言无关，无问题
turn_detection=EnglishModel(),  # 英文回合检测，中文场景不适用
```

核心问题：
1. **Deepgram** — 英文优先 STT，中文识别质量不如国内方案，海外 API 延迟高
2. **Cartesia** — 英文语音合成，中文效果差，海外 API 延迟高
3. **EnglishModel** — 英文回合检测，中文场景应用中文模型或 Multilingual
4. **silero.VAD** — 语言无关，可保留

在 25 秒时限场景下，STT/TTS 的延迟直接影响每轮响应速度，海外服务的不稳定延迟是硬伤。

#### 候选方案

| 组件 | 当前（英文） | 国内候选 | 特点 |
|------|-------------|---------|------|
| STT | Deepgram | 阿里云 Paraformer | 中文识别精度高，流式支持好，延迟低 |
| | | 讯飞语音识别 | 中文老牌，识别准确，但 API 风格偏旧 |
| | | 百度语音识别 | 价格低，精度中等 |
| TTS | Cartesia | 阿里云 CosyVoice | 中文合成自然度高，支持流式，延迟低 |
| | | 讯飞 TTS | 合成质量好，声音选择多 |
| | | 百度语音合成 | 价格低，质量中等 |
| Turn Detection | EnglishModel | MultilingualModel / 中文模型 | LiveKit 是否提供中文 turn detector 待确认 |

#### LiveKit 插件适配

LiveKit Agents 通过插件接口接入 STT/TTS。当前 `livekit-plugins-deepgram`、`livekit-plugins-cartesia` 是官方插件。国内方案可能没有现成 LiveKit 插件，需自写适配器（实现 LiveKit 的 STT/TTS 接口）。适配器工作量不大，核心是实现流式回调接口。

#### 选型维度

| 维度 | Deepgram+Cartesia | 国内方案（如阿里云） |
|------|-------------------|---------------------|
| 中文质量 | 差 | 好 |
| 延迟 | 海外调用高（100-300ms+） | 国内调用低（30-50ms） |
| LiveKit 插件 | 官方提供 | 可能需自写适配器 |
| 成本 | 按量计费 | 按量计费，价格相近 |
| 稳定性 | 受网络波动影响 | 国内网络稳定 |

#### 待确认

- 阿里云 Paraformer + CosyVoice 是否有现成 LiveKit 插件，或需自写适配器
- LiveKit 是否提供中文/Multilingual turn detector
- 是否考虑同时支持多 STT/TTS 提供商（通过配置切换，类似 `MODEL_PROVIDER` 模式）
